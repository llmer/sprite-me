# Animating Sprites with sprite-me

## ⚠️ Experimental — read before using

The current animation backend uses `batch_size=N` on a single FLUX prompt to produce N independent same-character candidate frames. Each candidate is a full centered character sprite, not a strip — you can use them as:

- **Idle-breathing variants**: small random pose/stance differences between frames
- **Candidate selection**: generate N, pick the 2–4 best hand-edited keyframes
- **Variation testing**: see what the LoRA does with your prompt across seeds

It is **NOT a proper animation cycle**. There is no pose control, no temporal coherence, no guarantee the frames flow naturally as a walk/attack sequence. For real motion cycles you'll need to either:

- Hand-pick frames and touch them up in a pixel editor
- Use a dedicated pose-conditioned approach (ControlNet OpenPose or Sprite Sheet Diffusion) which isn't wired in yet

If the user expects a game-engine-ready walk cycle from `animate_sprite`, warn them about this limitation first.

## Prerequisite: Asset Must Be In Manifest

`animate_sprite` only works on assets tracked in the sprite-me manifest. If the user has a local PNG that wasn't generated by sprite-me, you must import it first:

```
1. import_image(source="/path/to/player.png", name="player")
   -> asset_id = "abc123"

2. animate_sprite(asset_id="abc123", animation="walk")
```

**Never** try to pass a file path or data URL directly to `animate_sprite`.

## Animation Presets

Built-in presets with auto-enhanced prompts:

| Preset | What it generates | Typical frames |
|---|---|---|
| `idle` | Subtle breathing/standing loop | 4 |
| `walk` | Walk cycle, side view | 6-8 |
| `run` | Running cycle, faster than walk | 6-8 |
| `attack` | Generic attack animation | 6-8 |
| `jump` | Jump + landing | 6 |
| `death` | Falling/collapse | 6 |
| `cast` | Magic casting | 6-8 |

For anything else (or to customize a preset), use `custom_prompt`:

```
animate_sprite(
  asset_id="abc123",
  custom_prompt="dual-wielding katanas, whirlwind spin attack, blades visible",
  frames=8
)
```

## Frame Count Guidance

| Animation | Frames |
|---|---|
| Simple loop (idle, breathing) | 4 |
| Walk/run cycle | 6-8 |
| Attack (slash, punch) | 6-8 |
| Complex action (combo, cast, transformation) | 8-12 |

Don't use more than 12 frames — the model struggles to keep consistency across many frames in a single generation.

## Default Settings

| Setting | Default | Notes |
|---|---|---|
| `animation` | `"idle"` | Pick the closest preset, override with `custom_prompt` if needed |
| `frames` | `6` | See frame count table above |
| `edge_margin` | `6` | Prevents limbs from clipping at frame edges |
| `auto_enhance` | `true` | Expands short prompts into detailed motion descriptions |

## Writing Custom Motion Prompts

Ground motion prompts in the character's context — what are they doing, with what, how:

**Good:**
- `"knight raising greatsword overhead then swinging down"`
- `"archer drawing bow, aiming, releasing arrow"`
- `"wizard summoning glowing blue energy in both hands"`

**Bad:**
- `"attack"` (use the preset instead)
- `"moving"` (not specific)
- `"fight"` (too abstract, no clear motion)

## Output Format

`animate_sprite` returns a single sprite sheet PNG containing all frames in a row, plus individual frame PNGs saved separately. The returned `path` is the sheet; individual frames are at `assets/{asset_id}_frame{NN}.png`.

To use in a game engine:
- **Godot**: Import the sheet, set `hframes` to the frame count, create an `AnimatedSprite2D`
- **Unity**: Import the sheet, set Sprite Mode to "Multiple", slice into frames
- **Phaser**: Use `spritesheet` loader with `frameWidth`/`frameHeight`

## Common Mistakes

1. **Forgetting to import**: Passing a raw file path to `animate_sprite` → use `import_image` first
2. **Too many frames**: 15+ frames in one generation → consistency breaks down, use 6-8 instead
3. **Vague prompts**: `"cool animation"` → be specific about the motion
4. **Wrong preset**: Using `attack` for a magic spell → use `cast` or custom prompt
